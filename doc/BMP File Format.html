BMP File Format

History

The BMP bitmap file format, sometimes called DIB file format (for device‐independent bitmap), is an image file format used to store bitmap digital images, usually named with the file extension of .BMP or .DIB, especially on Microsoft Windows and IBM OS/2 operating systems.

The original bitmap format created for Windows 1.0 was very simple. It had a fixed color palette (the one in use by the graphics card), did not support bitmap data compression, and was designed to support the most popular IBM PC graphics cards in use at the time (MGA, Hercules, CGA, EGA and others). This defunct format is now often referred to as the original Windows device‐dependent bitmap (DDB).

As Windows 2.0 was being developed, EGA ruled the graphics cards and VGA started to replace it. So, support for a programmable color palette was added to both Windows and BMP file format, allowing user‐definable color data to be stored along with the bitmap data that used it. When stored in memory, this collection of information is known as a Microsoft Device Independent Bitmap, or DIB. When this information is written out to a file, it is known as the Microsoft Bitmap Format, or BMP. When you hear references to the DIB file format, it is BMP that is being referred to.

During the development of Windows BMP version 2, Microsoft shared responsibility with IBM for the development of early versions of IBM's OS/2 operating system. When IBM Presentation Manager, the OS/2 graphical user interface, required a bitmap format, the Windows BMP format was used. Thus, the Microsoft Windows 2.x and IBM OS/2 1.x BMP formats are identical.

The BMP format modified for Windows 3.0 differs only slightly from the IBM Presentation Manager bitmap format (OS/2 v2.x), that preceded it. Note that later revisions designed to support Presentation Manager 2.x and above, have resulted in further divergence between the Microsoft Windows and IBM OS/2 BMP file formats.

The structure of BMP format files is closely tied to the API of both Windows and OS/2. In this regard, BMP was never meant to be a portable format or used for bitmap data interchange between different operating systems. As each of these operating system APIs has changed, the BMP format has changed along with it.

There are currently four versions of BMP under Windows (2, 3 with NT and CE variations, 4 and 5) and two versions under OS/2 (1 and 2 with, six possible variations each).

All the BMP versions originated on Intel based machines and thus share a common little‐endian heritage. The current BMP format is otherwise hardware‐independent and can accommodate images with up to 32‐bit color (1, 4, 8, 16, 24 and 32 bit‐per‐color). Its basic design makes it a good general‐purpose format that can be used for color or black‐and‐white image storage if file size is not a factor. Its main virtues are its simplicity and widespread support in the PC marketplace.

BMP File Gap 1 and Gap 2

The Gap 1 and Gap 2 on the bitmap file format, are optional and so, they can be 0 bytes. When both gaps are set to zero (0), the bitmap is called a Packed DIB.
On packed DIBs in memory, the Windows Bitmap Pixels Array and OS/2 Bitmap Data, must start at a memory address that is a multiple of 4 bytes. In some cases, it may be necessary to adjust the number of entries in the color table in order to force the memory address alignment.
The optional ICC color correction profile data, should immediately follow the pixel array data, if used.

Note: Packed DIBs are required by Windows clipboard, patterned brush, resources APIs functions as well as others Win 32 APIs.

Gap 1 is for aligning the Windows Bitmap Pixels / OS/2 Bitmap Data values and Gap 2 the ICC (International Color Consortium) Color Correction Profile v1 data values, case it exists.

Their existence is for speeding the read and write operations to the file, by aligning the structures to a 16‐bit, 32‐bit or 64‐bit word (unsigned integer), according the OS platform CPU word. Aligning to 32‐bit (a DWORD on Win32), should produce the best results. For in-memory DIBs, is better to use packed DIBs, aka Gap 1 and Gap 2 equal to 0 bytes.

 

BMP File Windows DDB Bitmap Pixels, Windows Bitmap Pixels and OS/2 Bitmap Data

For Windows DDB Bitmap Pixels, the pixels values follow immediately after the File Header v1 + Windows DDB Header v1.
It consists of Height scan lines, with the size, in bytes of ByteWidth. Must be aligned to 8‐bit:     ByteWidth = ((Width * BitsPerPixel) / 8) + 1;
The total size of the Windows DDB Bitmap Pixels, can be calculated by the formula:     DdbSize = Height * ByteWidth);
Each pixel value occupies BitsPerPixel bits on the scan line, from left to right.
This value is an index to the Windows OS system pallet. The padding is ignored and normally is 0.

For Windows Bitmap Pixels and OS/2 Bitmap Data, the pixels values follow immediately after the Windows Color Table or OS/2 Color Palette + Gap 1 (the value pointed by BITMAPFILEHEADER.BitmapOffset).
It consists of Height scan lines, with the size, in bytes of ByteWidth. Must be aligned to 32‐bit:
    scanLineSize = 4 * (((BitmapHeader.Width * BitmapHeader.BitsPerPixel) + 31) / 32);
If Height < 0 then the scan lines are stored top‐down i.e., its origin is the upper left corner of the bitmap (0, 0). Else if Height > 0, then it's stored bottom‐up i.e., its origin is the lower left corner of the bitmap (0, Last-Y).
The total size of the Windows Bitmap Pixels or OS/2 Bitmap Data, can be calculated by the formula:
    DibSize = Height * scanLineSize);
Each pixel value occupies BitsPerPixel bits on the scan line, from left to right.
This value is an index to the bitmap pallet (for bits‐per‐pixel ≤ 8) or the pixel color (for bits‐per‐pixel ≥ 16).
The padding is ignored and normally is 0.

 

BMP File Header v1 + Windows DDB Header (10 Bytes)

This header/structure has created for Microsoft Windows 1.x bitmaps. All the integer values are stored in little‐endian format (i.e. least‐significant byte first). It has a total size of 10 bytes.

This block of bytes is at the start of the file and is used to identify the file as an BMP file and that it is not damaged.

DDB files contain only a 10-byte file header followed by uncompressed bitmap data.

The image data immediately follows the header and is stored in an uncompressed format. Each pixel stores an index value into the fixed system colormap used by Windows 1.0. The presence of scan line margin may be determined by comparing the calculated width of a line in bytes with the actual width of the line in bytes stored as the value of the ByteWidth field.

The minimum value of ByteWidth, can be calculated by the formula:
    ByteWidth = ((Width * BitsPerPixel) / 8) + 1;

BMP Windows Color Table v1 and OS/2 Color Palette v1 (3 Bytes)

This header/structure has created for Microsoft Windows 2.x and IBM OS/2 1.x bitmaps. All the integer values are stored in little‐endian format (i.e. least‐significant byte first). Can only be used with Windows DIB Header v2 and OS/2 Bitmap Header v1 (Windows BITMAPCOREHEADER or OS/2 OS21XBITMAPHEADER structures). It has a total size of 3 bytes.

This is the third header in the file, if present, and follows immediately the Windows DIB Header v2 and OS/2 Bitmap Header v1 (Windows BITMAPCOREHEADER or OS/2 OS21XBITMAPHEADER structures). Is used to store the BMP palette.

This header/structure is mandatory for bitmaps with less or equal than 8 bits‐per‐pixel (1, 4 and 8). It's optional for bitmaps with more than 8 bits‐per‐pixel (24)

The Color Palette/Table should be order by importance, i.e., most used colors first.

The maximum number of entries on the Color Palette/Table 2n (where n is the number of bits‐per‐pixel), can be calculated by the formula:
    MaxNumberOfEntries = 1 << BitsPerPixel;
To detect the presence of a color palette in a BMP file, use this formula: 


      int StartOfPalBytes = 0;
      int NumberOfEntries = 0;
      int MaxNumberOfEntries = 1 << BitmapCoreHeader.bcBitCount;
      StartOfPalBytes = sizeof(BITMAPFILEHEADER) + BitmapCoreHeader.bcSize;
      NumberOfEntries = (BitmapFileHeader.bfOffBits - StartOfPalBytes) / sizeof(RGBTRIPLE);
      if (NumberOfEntries < 0) {
          /* ERROR: No Color Pallet */
          NumberOfEntries = 0;
      }
      if (NumberOfEntries > MaxNumberOfEntries) {
          NumberOfEntries = MaxNumberOfEntries;
      }
      RGBTRIPLE ColorPallet_RGB[] = loadColorPallet_RGB(NumberOfEntries);

BMP Windows Color Table v2 and OS/2 Color Palette v2 (4 Bytes)

This header/structure has created for Microsoft Windows 3.x and IBM OS/2 2.x bitmaps. All the integer values are stored in little‐endian format (i.e. least‐significant byte first). Can only be used with Windows DIB Header v3/v4/v5 and OS/2 Bitmap Header v2 (Windows BITMAPINFOHEADER/BITMAPV4HEADER/BITMAPV5HEADER or OS/2 OS22XBITMAPHEADER structures). It has a total size of 4 bytes.

This is the third header in the file, if present, and follows immediately the Windows DIB Header v2 and OS/2 Bitmap Header v1 (Windows BITMAPCOREHEADER or OS/2 OS21XBITMAPHEADER structures). Is used to store the BMP palette.

This header/structure is mandatory for bitmaps with less or equal than 8 bits‐per‐pixel (1, 4 and 8). It's optional for bitmaps with more than 8 bits‐per‐pixel (24)

The Color Palette/Table should be order by importance, i.e., most used colors first.

The maximum number of entries on the Color Palette/Table 2n (where n is the number of bits‐per‐pixel), can be calculated by the formula:
    MaxNumberOfEntries = 1 << BitsPerPixel;
To detect the presence of a color palette in a BMP file, use this formula: 


      int StartOfPalBytes = 0;
      int NumberOfMasks = 0;
      int NumberOfMasksBytes = 0;
      int NumberOfEntries = 0;
      int MaxNumberOfEntries = 1 << BitmapInfoHeader.biBitCount;
      if (BitmapInfoHeader.biSize == sizeof(BITMAPINFOHEADER) {
          if (BitmapInfoHeader.biCompression == BI_BITFIELDS) {
              if (BitmapInfoHeader.biBitCount == 16) {
                  NumberOfMasks = 3;
              }
              else {
                  if (BitmapInfoHeader.biBitCount == 32) {
                      NumberOfMasks = 4;
                  }
              }
          }
          else {
              if (BitmapInfoHeader.biCompression == BI_ALPHABITFIELDS) {
                  NumberOfMasks = 4;
              }
          }
      }
      if (NumberOfMasks > 0) {
          NumberOfMasksBytes = NumberOfMasks * sizeof(RGBQUAD);
          BitmapInfoHeader.biSize = BitmapInfoHeader.biSize + NumberOfMasksBytes;
      }
      StartOfPalBytes = sizeof(BITMAPFILEHEADER) + BitmapInfoHeader.biSize;
      if (BitmapFileHeader.bfOffBits < StartOfPalBytes) {
          /* ERROR: No Bitfields masks */
          BitmapInfoHeader.biSize = BitmapInfoHeader.biSize - NumberOfMasksBytes;
          NumberOfMasks = 0;
          NumberOfMasksBytes = 0;
          StartOfPalBytes = sizeof(BITMAPFILEHEADER) + BitmapInfoHeader.biSize;
          /*
          Set default Bitfields masks for:
              BitmapInfoHeader.bV4RedMask,
              BitmapInfoHeader.bV4GreenMask,
              BitmapInfoHeader.bV4BlueMask,
              BitmapInfoHeader.bV4AlphaMask
          */
      }
      NumberOfEntries = (BitmapFileHeader.bfOffBits - StartOfPalBytes) / sizeof(RGBQUAD);
      if (NumberOfEntries < 0) {
          /* ERROR: No Color Pallet */
          NumberOfEntries = 0;
      }
      if ((BitmapInfoHeader.biClrUsed > MaxNumberOfEntries) || (BitmapInfoHeader.biClrUsed == 0)) {
          BitmapInfoHeader.biClrUsed = MaxNumberOfEntries;
      }
      if (NumberOfEntries < BitmapInfoHeader.biClrUsed) {
          BitmapInfoHeader.biClrUsed = NumberOfEntries;
      }
      else {
          BitmapInfoHeader.biClrUsed = (BitmapInfoHeader.biClrUsed == 0) ?  NumberOfEntries : BitmapInfoHeader.biClrUsed;
      }
      if (BitmapInfoHeader.biClrImportant < BitmapInfoHeader.biClrUsed) {
          BitmapInfoHeader.biClrImportant = (BitmapInfoHeader.biClrImportant == 0) ?  BitmapInfoHeader.biClrUsed : BitmapInfoHeader.biClrImportant;
      }
      else {
          BitmapInfoHeader.biClrImportant = (BitmapInfoHeader.biClrUsed == 0) ?  NumberOfEntries : BitmapInfoHeader.biClrUsed;
      }
      RGBQUAD ColorPallet_RGBA[] = loadColorPallet_RGBA(BitmapInfoHeader.biClrImportant);
