BMP File Format

History

The BMP bitmap file format, sometimes called DIB file format (for device-independent bitmap), is an image file format used to store bitmap digital images, usually named with the file extension of .BMP or .DIB, especially on Microsoft Windows and IBM OS/2 operating systems.

The original bitmap format created for Windows 1.0 was very simple. It had a fixed color palette, did not support bitmap data compression, and was designed to support the most popular IBM PC graphics cards in use at the time (MGA, Hercules, CGA, EGA and others). This defunct format is now often referred to as the original Windows device-dependent bitmap (DDB).

As Windows 2.0 was being developed, EGA ruled the graphics cards and VGA started to replace it. So, support for a programmable color palette was added to both Windows and BMP file format, allowing user-definable color data to be stored along with the bitmap data that used it. When stored in memory, this collection of information is known as a Microsoft Device Independent Bitmap, or DIB. When this information is written out to a file, it is known as the Microsoft Bitmap Format, or BMP. When you hear references to the DIB file format, it is BMP that is being referred to.

During the development of Windows BMP version 2, Microsoft shared responsibility with IBM for the development of early versions of IBM's OS/2 operating system. When IBM Presentation Manager, the OS/2 graphical user interface, required a bitmap format, the Windows BMP format was used. Thus, the Microsoft Windows 2.x and IBM OS/2 1.x BMP formats are identical.

The BMP format modified for Windows 3.0 differs only slightly from the IBM Presentation Manager bitmap format (OS/2 v2.x), that preceded it. Note that later revisions designed to support Presentation Manager 2.x and above, have resulted in further divergence between the Microsoft Windows and IBM OS/2 BMP file formats.

The structure of BMP format files is closely tied to the API of both Windows and OS/2. In this regard, BMP was never meant to be a portable format or used for bitmap data interchange between different operating systems. As each of these operating system APIs has changed, the BMP format has changed along with it.

There are currently four versions of BMP under Windows (2, 3 with NT and CE variations, 4 and 5) and two versions under OS/2 (1 and 2 with, six possible variations each).

All the BMP versions originated on Intel-based machines and thus share a common little-endian heritage. The current BMP format is otherwise hardware-independent and can accommodate images with up to 32-bit color (1, 4, 8, 16, 24 and 32 bit-per-color). Its basic design makes it a good general-purpose format that can be used for color or black-and-white image storage if file size is not a factor. Its main virtues are its simplicity and widespread support in the PC marketplace.

BMP File Header v1 + Windows DDB Header (10 Bytes)

This header/structure has created for Microsoft Windows 1.x bitmaps. All the integer values are stored in little-endian format (i.e. least-significant byte first). It has a total size of 10 bytes.

This block of bytes is at the start of the file and is used to identify the file as an BMP file and that it is not damaged.

DDB files contain only a 10-byte file header followed by uncompressed bitmap data.

The image data immediately follows the header and is stored in an uncompressed format. Each pixel stores an index value into the fixed system colormap used by Windows 1.0. The presence of scan line margin may be determined by comparing the calculated width of a line in bytes with the actual width of the line in bytes stored as the value of the ByteWidth field.

The minimum value of ByteWidth, can be calculated by the formula:
    ByteWidth = ((Width * BitsPerPixel) / 8) + 1;

Table 1 - Windows 1.x File and DDB Header structure
Offset# Size (Bytes) Type Field Name Purpose
0000h 2 16-bit unsigned FileType The magic number used to identify the BMP file (always 0).
0002h 2 16-bit unsigned Width The bitmap image width, in pixels (must be > 0).
0004h 2 16-bit unsigned Height The bitmap image height, in pixels (must be > 0).
0006h 2 16-bit unsigned ByteWidth The width of the bitmap in bytes, per scan line. It is assumed that this value will include the size of any scan line margin that is present.
0008h 1 8-bit unsigned Planes The number of color planes used to store the bitmap. This value is always 1.
000Ah 1 8-bit unsigned BitsPerPixel The number of bits-per-pixel.
The following valid values are possible:
  • 1 - 2 colors Bitmap (MGA/Hercules/Monochrome)
  • 4 - 16 colors Bitmap (CGA/EGA)
  • 8 - 256 colors Bitmap (VGA)

 

BMP File Header v2 (14 Bytes)

This header/structure has created for Microsoft Windows 2.x and IBM OS/2 1.x bitmaps. All the integer values are stored in little-endian format (i.e. least-significant byte first). It has a total size of 14 bytes.

This block of bytes is at the start of the file and is used to identify the file as an BMP file and that it is not damaged.

The first two bytes of the BMP file format are normally the character 'B' and then the character 'M' in 1-byte ASCII encoding.

Offset# Size (Bytes) BITMAPFILEHEADER OS2BMPFILEHEADER Purpose
Type Field Name Type Field Name
0000h 2 16-bit unsigned bfType 16-bit unsigned FileType The magic number (in ASCII) used to identify the BMP file.
The following valid values are possible:
  • BM (4D42h) - Windows or OS/2 Bitmap;
  • BA (4142h) - OS/2 Bitmap Array;
  • CI (4943h) - OS/2 Color Icon;
  • CP (5043h) - OS/2 Color Pointer;
  • IC (4349h) - OS/2 Icon;
  • PT (5450h) - OS/2 Pointer.
0002h 4 32-bit unsigned bfSize 32-bit unsigned FileSize The total size of the BMP file in bytes.
0006h 2 16-bit unsigned bfReserved1 16-bit unsigned XHotSpot Windows: reserved (must be 0).
OS/2: X coordinate of hotspot.
0008h 2 16-bit unsigned bfReserved2 16-bit unsigned YHotSpot Windows: reserved (must be 0).
OS/2: Y coordinate of hotspot.
000Ah 4 32-bit unsigned bfOffBits 32-bit unsigned BitmapOffset The offset, i.e. starting address, of the byte where the bitmap data can be found.

 

BMP Windows DIB Header v2 and OS/2 Bitmap Header v1 (12 Bytes)

This header/structure has created for Microsoft Windows 2.x and IBM OS/2 1.x bitmaps. All the integer values are stored in little-endian format (i.e. least-significant byte first). It has a total size of 12 bytes.

This is the second header in the file and follows immediately the File Header v2 (Windows BITMAPFILEHEADER or OS/2 OS2BMPFILEHEADER structures). Is used to describe the DIB: dimensions, bits-per-pixel, etc..

0006h 2 16-bit unsigned bcHeight 16-bit unsigned Height The bitmap image height, in pixels (must be > 0).
0008h 2 16-bit unsigned bcPlanes 16-bit unsigned NumPlanes The number of color planes (must be 1).

BMP Windows Color Table v1 and OS/2 Color Palette v1 (3 Bytes)

This header/structure has created for Microsoft Windows 2.x and IBM OS/2 1.x bitmaps. All the integer values are stored in little-endian format (i.e. least-significant byte first). Can only be used with Windows DIB Header v2 and OS/2 Bitmap Header v1 (Windows BITMAPCOREHEADER or OS/2 OS21XBITMAPHEADER structures). It has a total size of 3 bytes.

This is the third header in the file, if present, and follows immediately the Windows DIB Header v2 and OS/2 Bitmap Header v1 (Windows BITMAPCOREHEADER or OS/2 OS21XBITMAPHEADER structures). Is used to store the BMP palette.

This header/structure is mandatory for bitmaps with less or equal than 8 bits-per-pixel (1, 4 and 8). It's optional for bitmaps with more than 8 bits-per-pixel (24)

The Color Palette/Table should be order by importance, i.e., most used colors first.

The total number of entries on the Color Palette/Table 2n (where n is the number of bits-per-pixel), can be calculated by the formula:
    NumberOfEntries = 1 << BitsPerPixel;

Type Field Name Type Field Name
0000h 1 8-bit unsigned rgbtBlue 8-bit unsigned Blue The blue component intensity value of the RGB color (must be between 0 and 255).
0001h 1 8-bit unsigned rgbtGreen 8-bit unsigned Green The green component intensity value of the RGB color (must be between 0 and 255).
0002h 1 8-bit unsigned rgbtRed 8-bit unsigned Red The red component intensity value of the RGB color (must be between 0 and 255).

 

BMP OS/2 Bitmap Header v2 (16 Bytes to 64 Bytes)

This header/structure has created for IBM OS/2 2.x bitmaps. All the integer values are stored in little-endian format (i.e. least-significant byte first). It has a minimum size of 16 bytes and a maximal size of 64 bytes.

This is the second header in the file and follows immediately the File Header v2 (Windows BITMAPFILEHEADER or OS/2 OS2BMPFILEHEADER structures). Is used to describe the DIB: dimensions, bits-per-pixel, pallet size, compression, etc..

The fields above 16 bytes and below 64 bytes, that aren't on the file, are assumed to be 0.

BMP Windows Color Table v2 and OS/2 Color Palette v2 (4 Bytes)

This header/structure has created for Microsoft Windows 3.x and IBM OS/2 2.x bitmaps. All the integer values are stored in little-endian format (i.e. least-significant byte first). Can only be used with Windows DIB Header v3/v4/v5 and OS/2 Bitmap Header v2 (Windows BITMAPINFOHEADER/BITMAPV4HEADER/BITMAPV5HEADER or OS/2 OS22XBITMAPHEADER structures). It has a total size of 4 bytes.

This is the third header in the file, if present, and follows immediately the Windows DIB Header v2 and OS/2 Bitmap Header v1 (Windows BITMAPCOREHEADER or OS/2 OS21XBITMAPHEADER structures). Is used to store the BMP palette.

This header/structure is mandatory for bitmaps with less or equal than 8 bits-per-pixel (1, 4 and 8). It's optional for bitmaps with more than 8 bits-per-pixel (24)

The Color Palette/Table should be order by importance, i.e., most used colors first.

The total number of entries on the Color Palette/Table 2n (where n is the number of bits-per-pixel), can be calculated by the formula:
